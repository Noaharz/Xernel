# Xernel — Vollständiges Konzept & Schritt-für-Schritt-Roadmap

> **Arbeitsname:** Xernel
> **Version dieses Dokuments:** 0.1 (Konzept, 2026-05-18)
> **Inspirationsquelle:** `0.0.1_Gemini.txt` (Mikrokernel + Rust + Capabilities)
> **Vorbilder:** seL4, Redox OS, Fuchsia/Zircon, Genode, MINIX 3

---

## 0. Vision in einem Satz

> Xernel ist ein **moderner, in Rust geschriebener Mikrokernel** mit **capability-basierter Sicherheit**, der so klein ist, dass er **(perspektivisch) formal verifizierbar** wird, und der durch eine **POSIX-Kompatibilitätsschicht** trotzdem mit bestehender Software umgehen kann.

Drei nicht-verhandelbare Eigenschaften:

1. **Stabilität durch Isolation** — Treiber, Dateisysteme, Netzwerk-Stack laufen alle im User-Space. Crash eines Treibers ≠ Kernel-Panic.
2. **Sicherheit durch die Sprache** — Kein Buffer Overflow, kein Use-after-free im Kernel. Rust mit `#![forbid(unsafe_code)]` außerhalb klar markierter HAL-Bereiche.
3. **Sicherheit durch das Modell** — Kein „Root darf alles". Jeder Zugriff ist eine explizite, weitergebbare Capability (seL4-Stil).

## 1. Strategische Grundentscheidungen (müssen *vor* Phase 0 stehen)

| Entscheidung | Empfehlung für Xernel | Alternativen |
|---|---|---|
| Kernel-Typ | **Mikrokernel** (seL4-inspiriert) | Hybrid (wie macOS XNU) |
| Sprache (Kernel) | **Rust** (`no_std`, nightly) | Zig, C + Rust |
| Sprache (Userland) | Rust + C (für Ports) | nur Rust |
| Ziel-ISA Phase 1 | **x86_64** (am besten dokumentiert, QEMU-tauglich) | RISC-V als zweite Plattform |
| Ziel-ISA Phase 2+ | **AArch64** und **RISC-V (rv64gc)** | — |
| Bootloader | **Limine** (modern, einfach, BIOS+UEFI) | Eigener Bootloader, GRUB, `bootloader`-Crate |
| Build-System | **Cargo** + Custom Target JSON + xtask | Make, Meson |
| Test-Plattform | **QEMU** + **GDB** + KVM | echtes Hardware-Lab (später) |
| Lizenz | **MPL-2.0** *oder* **Apache-2.0 + MIT** | GPLv3 |
| ABI-Kompatibilität | Eigene Kernel-ABI (capability-basiert) + **POSIX-Layer in Userspace** | reine Linux-ABI nachbauen |
| Userland-libc | **relibc** (aus Redox) als Startpunkt, später eigene | musl-Port |
| Init / Root-Server | **eigener Root-Server** in Rust | systemd-ähnlich |
| Versionsschema | SemVer, Pre-1.0 bis Treiber + POSIX stehen | — |

> **Wichtig:** Diese Tabelle ist der einzige Punkt, an dem es noch wirklich „nur" um Geschmack geht. Ab Phase 0 ist alles Konsequenz aus diesen Entscheidungen.

## 2. Realitäts-Check (bitte einmal lesen)

Ein eigener Kernel von 0 bis „läuft auf echter Hardware mit Desktop" ist:

- **Größenordnung:** 100k–1M Zeilen Code für die ersten 5 Jahre, danach Wachstum.
- **Zeit (Solo, ernsthaft):** Bootloader → „Hello, Kernel" in 1–2 Wochen. Erster Multitask-Userspace in 6–12 Monaten. Brauchbares System mit Shell + FS + Netzwerk in 2–4 Jahren.
- **Treiber-Problem:** Bleibt das Hauptproblem. Strategie unten in Phase 4.
- **Realistisches Endziel für 1 Person:** Lauffähiges Hobby-OS mit Shell, einfachem FS, Netzwerk und ein paar User-Programmen — auf x86_64 unter QEMU. Das ist **schon enorm.**

Das ist kein Argument gegen das Projekt — es ist der wichtigste Sport für Systemprogrammierer. Aber der Plan unten ist so geschrieben, dass jede Phase **eigenständig wertvoll** ist und auch dann etwas Vorzeigbares zurücklässt, wenn du eine Phase später nicht weitermachst.

```
├── userland/
│   ├── relibc-port/
│   ├── xsh/                    # Shell
│   └── coreutils/
├── docs/
│   ├── design/                 # ADRs (Architecture Decision Records)
│   └── book/                   # mdBook „The Xernel Book"
└── tests/                      # Integrationstests in QEMU
```

> **Faustregel:** Was nicht im Ordner `kernel/` liegt, läuft im User-Space.

## 4. Schritt-für-Schritt-Roadmap

### Phase 0 — Fundament & Toolchain (Woche 1)

**Ziel:** Lokal `cargo xtask run` startet QEMU mit einem Image, das „Hello, Xernel" auf die serielle Konsole schreibt.

1. Repository anlegen (`git init`, MIT/Apache-Lizenzdateien, README).
2. `rust-toolchain.toml` mit fixiertem nightly festschreiben (z. B. `nightly-2026-05-01`).
3. `x86_64-xernel.json` als Custom-Target schreiben (no_std, kein redzone, soft-float, kein std).
4. Bootloader **Limine** ins Repo einbinden, `limine.cfg` schreiben.
5. `xtask`-Crate: baut Kernel, packt mit Limine in ein ISO/IMG, startet QEMU mit `-serial stdio -no-reboot -no-shutdown -d int -M q35`.
6. Minimaler `kernel/src/main.rs`: `#![no_std]`, `#![no_main]`, Limine-Header, Panic-Handler, `loop {}`.
7. Serieller Treiber (16550 UART), `println!`-Makro.
8. **Meilenstein 0.1:** „Hello, Xernel" auf Serial; reproduzierbarer Build.

### Phase 1 — Bare-Metal Bootstrap (Monat 1–2)

**Ziel:** Der Kernel managt seinen eigenen Speicher und kann Interrupts.

1. GDT mit Kernel/User-Code/Data + TSS.
2. IDT mit Stub-Handlern für alle CPU-Exceptions; saubere `#PF`-, `#GP`-, `#DF`-Routinen mit Stack-Dump.
3. Frame-Allocator (Bitmap oder Buddy) auf Basis der Limine-Memory-Map.
4. 4-Level-Paging (x86_64) — eigene `PageTable`-Abstraktion, höhere-Halbinsel-Mapping des Kernels.
5. Heap im Kernel (`linked_list_allocator` oder `talc`) auf gemapptem Bereich.
6. Local APIC + IO-APIC initialisieren, PIC abschalten.
7. **`unsafe`-Budget einführen:** jede `unsafe`-Stelle bekommt einen `// SAFETY:`-Kommentar und wird in einem Audit-Log erfasst.
8. **Meilenstein 1.0:** Kernel kann Speicher allokieren, fängt Page-Faults sauber ab, druckt Backtrace.

### Phase 2 — Mikrokernel-Kern: Threads, Scheduler, Capabilities, IPC (Monat 3–6)

**Ziel:** Das eigentliche Herz von Xernel. Hier wird entschieden, ob das Design trägt.

1. **Capability-System** (seL4-inspiriert):
   - `CNode` (Capability-Tabelle) als kernel-managed Objekt.
   - Cap-Typen: `Untyped`, `Frame`, `PageTable`, `Thread`, `Endpoint`, `Notification`, `IrqHandler`, `IoPort`.
   - Jeder Syscall ist eine `invoke(cap, method, args)`-Operation. Es gibt keine globalen Privilegien.
2. **Untyped Memory** als einziger Allokationsmechanismus für Userspace: ein Server bekommt einen Untyped-Block und retypisiert ihn selbst in `Frame`s, `CNode`s usw.
3. **Threads + Address Spaces** als Cap-Objekte (`TCB`, `VSpace`).
4. **Scheduler:** zunächst Round-Robin mit fester Prioritätsstufe, später MLFQ. Preemption über LAPIC-Timer.
5. **IPC:**
   - Synchron: `Endpoint`-basierter Rendezvous (Send/Receive/Call/Reply) wie in seL4.
   - Asynchron: `Notification` (Bit-Set, ähnlich semaphores).
   - Shared Memory: über `Frame`-Caps, die in zwei VSpaces gemappt werden.
6. **Interrupts** werden als IPC zugestellt: ein User-Treiber registriert eine `IrqHandler`-Cap auf eine `Notification`.
7. Strenge Tests: jede Cap-Operation hat einen Unit-Test (in QEMU mit kleinem In-Kernel-Testrunner).
8. **Meilenstein 2.0:** Zwei Kernel-interne Threads tauschen über IPC Nachrichten aus, ein dritter wird vom Timer geweckt.

### Phase 3 — User-Space wird real (Monat 6–9)

**Ziel:** Code, der nicht im Kernel liegt, läuft und kommuniziert mit dem Kernel ausschließlich über Capability-Invocations.

1. ELF-Loader im Root-Server (nicht im Kernel!).
2. Erster User-Prozess: `init` schreibt „Hello from userspace" über eine `Endpoint`-Cap an einen Konsolen-Server.
3. **Root-Server (`rootserver`):**
   - Wird vom Kernel mit *allen* Untyped-Caps gestartet.
   - Erzeugt initiale Dienste (Process Manager, Console Driver, VFS-Stub).
   - Reicht Caps an Kinder weiter — nur die, die sie brauchen.
4. `xstd` (User-Space-Standard-Crate): sichere Wrapper um die rohen Syscall-Stubs aus `xabi`.
5. Saubere Fehlersemantik: alles ist `Result<T, CapError>`.
6. **Meilenstein 3.0:** `init` + `console` + `pm` laufen als getrennte User-Prozesse; `init` startet `pm`, `pm` startet `xsh-stub`.

### Phase 4 — Treiber-Framework & erste Treiber (Monat 9–15)

**Ziel:** Echte Hardware (oder QEMU-Hardware) tut etwas Nützliches — im User-Space.

1. **PCI/PCIe-Enumerator** als User-Service (`pci`).
2. **virtio**-Treiber zuerst (virtio-blk, virtio-net, virtio-gpu) — einfach, gut dokumentiert, läuft in QEMU.
3. Erst danach „echte" Treiber: NVMe, xHCI (USB), AHCI, e1000.
4. **Treiber-Framework `xdriver`:**
   - DMA-Buffer als `Frame`-Cap mit physischer Adresse.
   - MMIO als `Frame`-Cap mit `device_memory`-Attribut.
   - IRQ als `IrqHandler`-Cap → `Notification`.
   - Hot-Restart: stirbt ein Treiber, startet der `pm` ihn neu, der `vfs` reconnected den Endpoint. **Das ist die Stabilitätsversprechung aus der Vision.**
5. **Meilenstein 4.0:** Lesen einer Datei (FAT32) von virtio-blk; Ping auf virtio-net.

### Phase 5 — System-Services (Monat 15–24)

**Ziel:** Aus „läuft" wird „benutzbar".

1. **VFS-Server** (`vfs`) — implementiert Mount-Namespace, leitet I/O an FS-Server.
2. Dateisystem-Server: zuerst **FAT32** und **ext2** (zum Lesen von vorbereiteten Images), dann **eigenes Xernel-FS** mit CoW + Snapshots.
3. **Netz-Stack** als User-Service auf Basis von **`smoltcp`**.
4. **Console-Multiplexer** / TTY-Server.
5. **Process Manager** mit echter Prozessverwaltung (fork-Äquivalent via cap-cloning).
6. **Meilenstein 5.0:** Shell mit Pipes, `ls`, `cat`, `wget`-Äquivalent gegen einen echten HTTP-Server.

### Phase 6 — Userland & Kompatibilität (Monat 18–36, parallel zu 5)

**Ziel:** Nicht jedes Programm muss neu geschrieben werden.

1. **POSIX-Kompatibilitäts-Layer** als Library OS:
   - `relibc` (aus Redox) portieren — übersetzt POSIX-Syscalls in Xernel-Cap-Invocations.
   - Damit lässt sich ein großer Teil von `coreutils`, `busybox`, `bash`, `vim`, sogar `gcc/clang` portieren.
2. **Eigene Shell `xsh`** in Rust.
3. **Paketformat** und einfacher Paketmanager (`xpkg`).
4. **Meilenstein 6.0:** `bash` + `coreutils` + `vim` laufen.

### Phase 7 — Sicherheit, Härtung, Verifikation (laufend ab Phase 2, intensiv ab Monat 24)

1. **Static-Analysis-Gate in CI:** `clippy::pedantic`, `miri` für Kernel-Tests in einer Sim-Variante, `cargo-audit`, `cargo-deny`.
2. **`unsafe`-Budget** veröffentlichen und nur nach Review wachsen lassen.
3. **Side-Channel-Härtung:** KPTI-Äquivalent (in Mikrokernel weniger relevant, aber Spectre-v2-Mitigations für IPC-Pfade).
4. **Secure Boot**: Limine + signiertes Kernel-Image.
5. **Capability-Audit-Tool:** statisches Werkzeug, das aus dem Root-Server-Setup einen Graphen zieht („wer hat welche Cap?").
6. **Optionales Fernziel:** ausgewählte Kernel-Kernpfade mit Kani oder Prusti verifizieren. Volle seL4-Style-Beweise wahrscheinlich nicht realistisch, **Teilverifikation** dagegen sehr wohl.

### Phase 8 — Distribution & Community (ab Monat 30)

1. **Installer-ISO** (Live + Install).
2. Eigene Webseite, Forum/Matrix, Issue-Tracker.
3. Versionspolitik: 0.x = Hobby, 1.0 = stabile Kernel-ABI.
4. **Hardware-Liste**: ehrlich dokumentieren, was läuft.

## 5. Querschnittsthemen (laufen durch alle Phasen)

- **ADRs (`docs/design/000X-*.md`)** — jede signifikante Entscheidung wird als kurzer Architecture-Decision-Record festgehalten. Zukünftiges Du wird dir danken.
- **CI von Tag 1** — GitHub Actions o. ä.: baut Kernel, startet QEMU, fährt Test-Suite, lehnt Merges bei Fail ab.
- **Testpyramide** — Unit-Tests pro Modul, In-Kernel-Tests in QEMU, Integrationstests „bootet ein Userland", später Fuzzing der Syscall-Oberfläche.
- **Reproduzierbare Builds** — pinned toolchain, `Cargo.lock` committed, deterministische Image-Erzeugung.
- **`docs/book/`** — `mdBook` „The Xernel Book", wächst mit dem System.

## 8. Offene Fragen, die du beantworten solltest

- **Plattform-Priorität:** Nur x86_64? Oder von Anfang an auch AArch64 (Apple Silicon, Raspberry Pi)?
- **Lizenz:** MIT/Apache (maximal liberal, Community-freundlich) oder MPL-2.0 (Copyleft pro Datei)?
- **Kompatibilitäts-Ehrgeiz:** Reicht POSIX-Subset, oder soll auch Linux-Binary-Kompatibilität (wie WSL1) Ziel sein? Letzteres ist ein eigenes Großprojekt.
- **Team:** Solo? Mit Freunden? Offen Open Source ab Tag 1 oder erst ab 0.3?
- **Hardware:** Soll Xernel irgendwann auf deinem eigenen Rechner booten, oder bleibt es bewusst „QEMU only"?

> Wenn du diese fünf Fragen beantwortest, kann ich diesen Plan zu einem konkreten Issue-Backlog für Phase 0 + 1 verdichten und dir das Repo scaffolden.
